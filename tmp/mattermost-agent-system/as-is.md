# AS-IS: текущая ручная схема агентной разработки

Этот документ описывает текущий рабочий процесс, в котором владелец управляет несколькими сессиями Codex через Telecodex/Telegram и GitHub. Цель документа — зафиксировать работающую ручную модель, чтобы потом перенести её в отдельную Mattermost-систему без потери важных деталей.

## Общая картина

Сейчас работа организована как ручная диспетчеризация нескольких независимых Codex-сессий на одном сервере. Владелец общается с менеджерской сессией через Telegram, получает от неё промпты для отдельных агентов, копирует эти промпты в соответствующие сессии и затем принимает результаты через GitHub Pull Request.

Mattermost в этой схеме отсутствует. Центральной системы для регистрации агентов, репозиториев, токенов, ролей, flow и статусов пока нет. Координация держится на договорённостях, именовании директорий, веток, PR и ручной памяти менеджера.

Ключевая особенность текущего процесса: он уже доказал, что модель "менеджер + несколько пар DEV/QA + короткие PR + владелец как финальный gate" работает практически. Но вся диспетчеризация пока выполняется вручную.

## Участники

- Владелец: принимает решения, сливает PR, запускает следующие шаги, утверждает или отклоняет результаты.
- Менеджерская Codex-сессия: анализирует состояние проекта, раздаёт задачи агентам, формирует промпты, отслеживает блокеры и рекомендует следующий шаг.
- DEV-агенты: выполняют разработку в отдельных клонах репозитория.
- QA/reviewer-агенты: работают в тех же директориях, что и соответствующие DEV-агенты, проверяют PR, оставляют замечания от имени владельца и ставят approval, если замечаний нет.
- GitHub: источник задач, PR, review, approvals и merge-фактов.
- Telegram/Telecodex: канал общения владельца с текущей Codex-сессией.

## Локальная организация

На сервере создано несколько отдельных клонов одного репозитория:

- `/home/s/projects/kodex` — менеджерская директория.
- `/home/s/projects/kodex-package-hub` — агентская директория для одного направления.
- `/home/s/projects/kodex-project-catalog` — агентская директория для другого направления.
- `/home/s/projects/kodex-provider-hub` — агентская директория для третьего направления.
- `/home/s/projects-second/kodex-agent-1` — дополнительный агентский клон.
- `/home/s/projects-second/kodex-agent-2` — дополнительный агентский клон.
- `/home/s/projects-second/kodex-agent-3` — дополнительный агентский клон.

В каждой агентской директории владелец может запускать две Codex-сессии:

- DEV-сессия, которая пишет код или документацию.
- QA-сессия, которая проверяет работу DEV-сессии в той же директории и в той же ветке.

Это отдельные клоны, а не git worktree. Поэтому агенты меньше мешают друг другу локально, но синхронизация `main`, submodule, env и локальных правил остаётся ручной.

Каждый клон выступает как рабочая зона конкретного агента. Агент может держать свою ветку, свои временные файлы, свой локальный контекст и свои результаты проверок. Менеджерская сессия не должна выполнять доменную работу в этих клонах, а только координирует, кому и что поручить.

## Конфигурация и секреты

Для работы агентам локально копируются или доступны:

- `.local/agents-temp-dev-rules.md` — временные правила работы агента на сервере.
- `bootstrap/host/config.env` — локальный файл с токенами и параметрами окружения.
- GitHub токены владельца и бота.
- Параметры доступа к live-кластеру и deploy tooling.

Секреты не должны попадать в PR, логи и ответы. Но сама раздача контекста по агентским директориям сейчас выполняется вручную.

## Как запускается работа

1. Владелец пишет менеджерской сессии, что сделано и что нужно дальше.
2. Менеджерская сессия подтягивает `main`, анализирует последние merge, открытые PR, документы и блокеры.
3. Менеджерская сессия решает, какого агента куда направить.
4. Менеджерская сессия выдаёт короткий промпт для конкретного агента.
5. Владелец копирует промпт в нужную DEV-сессию.
6. DEV-агент обновляет `main`, создаёт ветку, выполняет срез, открывает PR.
7. Владелец сообщает менеджеру ссылку на PR.
8. Менеджер выдаёт промпт для QA/reviewer-сессии.
9. QA-агент проверяет PR, оставляет замечания в GitHub или ставит approval.
10. DEV-агент исправляет замечания.
11. После approval владелец сливает PR.
12. Менеджер снова подтягивает `main`, анализирует состояние и выдаёт следующий промпт.

Важная часть процесса — промпт всегда содержит рабочую директорию, роль агента, целевой PR или задачу, ограничения по секретам, требования к проверкам и ожидаемый результат. Это фактически ручной аналог будущей карточки запуска агента.

## Текущий цикл разработки

Работа идёт короткими срезами. Срез считается завершённым, когда:

- DEV-агент открыл PR.
- QA-агент проверил PR.
- Все замечания устранены или обоснованно отклонены.
- Владелец слил PR.
- Менеджер обновил картину состояния.

Для крупных направлений менеджер старается не давать нескольким агентам пересекающиеся файлы, чтобы снизить конфликты. Если есть критический live-путь, менеджер может остановить параллельную разработку и направить одного агента на узкий блокер.

Если агент находит противоречие в документах, контрактах или live-состоянии, он останавливается и предлагает варианты. Менеджер или владелец выбирает вариант, после чего агент продолжает. Это правило важно сохранить в будущей системе: агент не должен молча обходить контракт.

## Работа с PR

Обычно DEV-агент:

- создаёт ветку;
- делает один логичный срез;
- запускает релевантные проверки;
- пушит ветку;
- открывает PR от имени бота;
- пишет, что сделано и какие проверки прошли.

QA-агент:

- работает в той же директории;
- читает актуальные правила и документацию;
- смотрит diff;
- оставляет inline-комментарии от имени владельца;
- ставит request changes, если есть блокеры;
- ставит approval от имени владельца, если замечаний не осталось.

Владелец вручную принимает решение о merge.

После merge владелец сообщает менеджеру, какие PR слиты. Менеджер подтягивает `main`, анализирует новые изменения и пересобирает план. В текущей схеме именно этот шаг заменяет автоматическое обновление состояния в системе управления агентами.

## Текущая работа с live-кластером

Отдельный агент обычно отвечает за deploy и live-проверки. Он:

- обновляет `main`;
- запускает штатные deploy-команды;
- проверяет готовность Kubernetes workloads;
- проверяет сертификаты и публичный контур;
- проверяет GitHub webhooks;
- проверяет диагностические gRPC/read surface;
- не создаёт PR, если код не менялся.

Такой агент работает как операционный исполнитель. Он сообщает факты: что раскатано, какие проверки прошли, где цепочка остановилась. Менеджер на основании этого решает, какой кодовый срез нужен дальше.

## Роль менеджера

Менеджерская сессия не должна писать доменный код без отдельной команды. Её роль:

- держать карту активных агентов;
- понимать, какой агент за какой домен отвечает;
- отслеживать слияния PR;
- подтягивать `main`;
- выявлять блокеры;
- решать, кого можно запускать параллельно;
- выдавать промпты DEV и QA агентам;
- объяснять владельцу текущий статус и следующий шаг.

## Что работает хорошо

- Можно параллелить работу по нескольким агентам.
- GitHub остаётся источником правды по PR, review и merge.
- Каждая задача идёт небольшим срезом.
- QA-сессия может независимо проверять DEV-сессию.
- Владелец сохраняет контроль над merge и approval.

## Основные проблемы

- Владелец вручную копирует промпты между чатами.
- Нет единого интерфейса статусов агентов, задач, PR и review-loop.
- Нет реестра агентов, ролей, промптов, flow и доступов.
- Нет автоматического запуска QA после DEV.
- Нет автоматического повторного запуска DEV после request changes.
- Нет централизованного управления токенами GitHub, OpenAI и репозиториев.
- Нет понятной модели подключения новых репозиториев.
- Нет единого аудита действий агентов.
- История решений размазана между Telegram, локальными сессиями, GitHub и памятью менеджера.
- Сложно быстро понять, какой агент чем занят и где заблокирован.
- Нет нативных кнопок действий: approve, request changes, rerun, stop, assign, summarize.
- Нет единого машинного журнала: кто запустил агента, с каким prompt, какой моделью, в какой директории, с какими правами и каким итогом.
- Нет безопасной формы для ввода токенов и ключей.
- Нет автоматической проверки, что агент работает в правильном репозитории, ветке и роли.
- Нет встроенного механизма "DEV исправляет замечания QA до трёх попыток, затем эскалация владельцу".
- Нет общей карточки run, где видны исходная задача, текущий шаг, PR, review status, блокер и следующие действия.

## Что нужно перенести в будущую систему

В Mattermost-систему нужно перенести не просто чат с агентом, а весь текущий рабочий контур:

- менеджерскую диспетчеризацию;
- пары DEV/QA;
- отдельные рабочие зоны агентов;
- короткие PR;
- обязательный review-loop;
- финальное решение владельца;
- безопасную работу с токенами;
- связь сообщений, PR, комментариев, проверок и live-операций;
- возможность остановиться при противоречии и запросить решение владельца.

## Практический вывод

Текущая схема доказала полезную рабочую модель: менеджер, несколько DEV/QA пар, короткие PR, GitHub как источник правды и владелец как финальный gate. Но она слишком ручная. Mattermost-система должна сохранить эту модель, но убрать ручное копирование промптов, ручную диспетчеризацию и ручное управление состоянием.
