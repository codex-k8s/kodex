# TO-BE: Mattermost-система для управления агентной разработкой

Этот документ описывает целевую идею отдельной системы, которая позволит управлять агентами из Mattermost примерно так же, как сейчас владелец управляет Codex через Telegram, но удобнее, структурированнее и с автоматизированными flow.

## Цель

Нужна система, в которой Mattermost становится рабочей поверхностью для агентной разработки:

- владелец и команда работают в каналах и тредах Mattermost;
- репозитории, токены, модели, агенты, роли и flow настраиваются через команды, формы и кнопки;
- агенты запускаются из сообщений Mattermost;
- PR, review и request changes автоматически связываются в один workflow;
- человек принимает финальные решения, а система автоматизирует повторяемую маршрутизацию.

Система не должна быть просто чат-ботом. Она должна стать минимальным операционным контуром для агентной разработки: кто может запустить агента, в каком репозитории, с каким prompt, от какого GitHub-аккаунта, с каким OpenAI-аккаунтом, в каком flow, с какими лимитами и как результат попадёт к человеку на принятие.

## Базовый пользовательский сценарий

1. Администратор поднимает Mattermost и подключает систему агентной разработки.
2. Система создаёт дефолтных ботов и служебные каналы.
3. Администратор добавляет GitHub organization или отдельные репозитории.
4. Администратор добавляет токены для доступа к репозиториям и bot-account в GitHub.
5. Администратор добавляет один или несколько OpenAI аккаунтов или ключей.
6. Администратор создаёт профили агентов: разработчик, reviewer, QA, менеджер, документатор.
7. Для каждого агента настраиваются инструкции: основной prompt, prompt для исправления замечаний, prompt для review.
8. Администратор создаёт минимальный flow: разработка, review, исправления, повторный review, финальное решение человека.
9. Пользователь пишет задачу в Mattermost.
10. Система создаёт run, запускает нужного агента, отслеживает PR, запускает reviewer, возвращает результат человеку.

## Принцип работы

Mattermost используется как основной интерфейс:

- сообщения пользователя становятся входом для агента или flow;
- тред Mattermost становится карточкой выполнения;
- кнопки в сообщениях заменяют ручное копирование промптов;
- каналы показывают статус проектов, агентов и блокеров;
- GitHub остаётся источником правды по коду, PR, review и merge;
- система хранит только безопасные метаданные, ссылки, статусы и журнал действий.

Агенты могут работать локально на сервере, в контейнерах или в Kubernetes-слотах. Для MVP достаточно одного сервера и одного способа запуска, но модель должна сразу допускать несколько runner-пулов.

## Основные сущности

### Workspace

Логическая область установки системы. В ней живут команды Mattermost, подключённые репозитории, токены, агенты и flow.

### Repository

Подключённый GitHub/GitLab репозиторий. Для него хранятся:

- provider;
- owner/name;
- default branch;
- правила веток;
- доступные bot tokens;
- разрешённые операции;
- политики создания PR;
- правила review и merge.

Репозиторий должен иметь статус подключения: не проверен, активен, ошибка доступа, webhook не настроен, токен устарел. Система должна уметь проверить, что бот может читать репозиторий, создавать ветки, открывать PR и оставлять review.

### Credential

Секрет или ссылка на секрет:

- GitHub user token;
- GitHub bot token;
- OpenAI API key;
- Mattermost bot token;
- webhook secret;
- deploy token.

Секреты не должны попадать в сообщения Mattermost, логи, PR или prompt. Пользователь видит только безопасные названия, статусы и scope.

Для каждого секрета нужна политика:

- кто может создать;
- кто может использовать;
- какие агенты имеют доступ;
- какие репозитории имеют доступ;
- как проверить работоспособность без вывода значения;
- когда секрет был последний раз использован;
- когда нужна ротация.

### OpenAI Account

Профиль доступа к OpenAI:

- название;
- model policy;
- лимиты;
- доступные модели;
- ключ или ссылка на секрет;
- правила использования агентами.

### Agent Profile

Профиль агента:

- имя;
- роль;
- описание;
- основной prompt;
- prompt для исправления замечаний;
- prompt для review;
- разрешённые репозитории;
- разрешённые credentials;
- модель;
- лимиты времени;
- лимиты попыток;
- режим работы: developer, reviewer, manager, QA, ops.

Профиль агента не равен конкретной запущенной сессии. Профиль описывает роль и правила запуска, а run создаёт конкретное выполнение. Один профиль может использоваться в разных flow и репозиториях.

Для каждого профиля желательно хранить:

- developer prompt;
- review prompt;
- fix prompt;
- правила остановки;
- правила работы с GitHub;
- список разрешённых инструментов;
- список запрещённых действий;
- требования к финальному отчёту.

### Flow

Шаблон процесса. Flow состоит из шагов:

- какой агент запускается;
- какой prompt используется;
- откуда берётся входной текст;
- что считается результатом;
- какой следующий шаг;
- что делать при request changes;
- сколько повторов разрешено;
- когда нужен человек.

Flow должен быть редактируемым без изменения кода. В MVP это может быть YAML/JSON-конфигурация или форма Mattermost, позже — отдельный UI. Главное, чтобы flow явно описывал последовательность шагов, условия переходов и лимиты повторов.

### Run

Конкретное выполнение flow или одного агента. Run связан с:

- Mattermost channel/thread;
- исходным сообщением;
- repository;
- branch;
- PR;
- agent profile;
- статусом;
- попытками;
- review result;
- финальным решением человека.

Run должен иметь один главный Mattermost thread. В нём владелец видит весь ход работы: кто запущен, что сделал, где PR, какие замечания, какой следующий шаг и какие действия доступны.

### Step

Один шаг run:

- developer step;
- reviewer step;
- fix step;
- QA step;
- owner approval step.

### Artifact

Результат работы:

- branch;
- commit;
- PR;
- review comments;
- summary;
- test report;
- ссылки на логи;
- решение владельца.

Для артефактов нужно хранить только безопасные ссылки и summary. Полные diff, файлы, секреты и сырые ответы провайдеров остаются в исходных системах.

## Архитектура MVP

Минимальная система может состоять из следующих компонентов:

- Mattermost Bot Adapter — принимает slash-команды, кнопки, сообщения и события.
- Orchestrator — хранит run, step, flow, состояния и принимает решения о следующем шаге.
- Agent Runner — запускает Codex/агента с нужным prompt и рабочим каталогом.
- GitHub Adapter — создаёт ветки, PR, comments, review и читает статусы.
- Credential Store — хранит ссылки на секреты и безопасные метаданные.
- OpenAI Account Router — выбирает OpenAI-аккаунт и модель для агента.
- Audit Log — фиксирует, кто что запустил и какой результат получил.
- Admin Commands — набор команд Mattermost для настройки репозиториев, токенов, профилей и flow.

Для MVP допустимо, чтобы это был один сервис с внутренними модулями. Но границы модулей лучше сразу держать явными, чтобы позже вынести их в отдельные сервисы.

### Административный канал

В административном канале настраиваются:

- репозитории;
- токены;
- OpenAI-аккаунты;
- профили агентов;
- flow;
- глобальные лимиты;
- default reviewers;
- права пользователей.

### Проектный канал

В проектном канале пользователь может:

- написать задачу обычным текстом;
- выбрать репозиторий;
- выбрать flow;
- запустить агента;
- увидеть текущие run;
- получить уведомление о PR или блокере.

### Тред выполнения

Тред выполнения должен показывать:

- исходное сообщение;
- выбранный flow;
- текущий step;
- активного агента;
- ссылку на branch/PR;
- summary работы;
- review status;
- кнопки действий;
- ошибки и блокеры;
- итоговое решение.

## Дефолтные боты Mattermost

### Manager Bot

Главный бот-диспетчер. Он:

- принимает команды;
- создаёт run;
- выбирает flow;
- сообщает статус;
- запускает агентов;
- показывает кнопки действий;
- просит человека принять решение.

Manager Bot должен быть единственным ботом, который принимает управленческие команды. Остальные боты могут быть видимыми исполнителями, но маршрутизацию лучше держать в одном месте.

### Developer Bot

Бот, от имени которого запускаются developer agents. Он может:

- создать ветку;
- внести изменения;
- открыть PR;
- исправить замечания.

### Reviewer Bot

Бот для review. Он:

- читает PR;
- сверяется с правилами;
- оставляет comments;
- ставит request changes или approval;
- пишет summary в Mattermost.

### QA Bot

Бот для проверок:

- запускает тесты;
- проверяет сборку;
- анализирует логи;
- пишет отчёт.

### Ops Bot

Бот для deploy и окружения:

- проверяет окружение;
- запускает deploy-команды;
- следит за статусами;
- докладывает о проблемах.

## Mattermost интерфейс

### Каналы

Минимальный набор:

- `agents-control` — управление системой.
- `agents-runs` — лента запусков.
- `repo-<name>` — рабочий канал репозитория.
- `agent-alerts` — ошибки и блокеры.

### Треды

Каждый run живёт в отдельном Mattermost thread. В thread попадает:

- исходная задача;
- выбранный flow;
- текущий статус;
- сообщения агентов;
- ссылки на PR;
- review results;
- кнопки действий.

### Команды

Примеры команд:

- `/agents repo add github codex-k8s/kodex`
- `/agents token add github-bot`
- `/agents openai add primary`
- `/agents profile create developer`
- `/agents flow create dev-review`
- `/agents run dev-review codex-k8s/kodex <текст задачи>`
- `/agents status`
- `/agents stop <run-id>`
- `/agents rerun <step-id>`

Для MVP можно начать с команд, а не с полноценного UI. Но команды должны возвращать карточки с кнопками, чтобы пользователь не запоминал длинный синтаксис.

### Кнопки

В сообщениях бота нужны кнопки:

- Запустить агента.
- Запустить review.
- Запросить изменения.
- Повторить исправление.
- Принять.
- Остановить.
- Показать логи.
- Открыть PR.
- Назначить reviewer.

Кнопки должны быть контекстными. Например, кнопка "Принять" появляется только когда reviewer уже поставил approval или когда владелец явно решил принять риск.

## Минимальный flow разработки

### Flow: developer-review-loop

1. Пользователь пишет задачу в Mattermost.
2. Manager Bot создаёт run.
3. Developer Agent получает:
   - исходный текст задачи;
   - repository context;
   - developer prompt;
   - правила проекта.
4. Developer Agent создаёт branch, коммит и PR.
5. Reviewer Agent запускается автоматически.
6. Reviewer Agent проверяет PR.
7. Если reviewer ставит approval:
   - run переходит к human approval.
8. Если reviewer делает request changes:
   - система запускает Developer Agent с fix prompt;
   - агент получает review comments;
   - агент пушит исправления в тот же PR;
   - reviewer запускается повторно.
9. Цикл повторяется от одного до трёх раз.
10. Если после лимита попыток approval нет:
   - run переходит в blocked;
   - человек получает summary и варианты действий.
11. Если approval есть:
   - человек принимает или отклоняет работу.

### Поведение при request changes

Если reviewer оставил request changes:

- система собирает только актуальные нерешённые замечания;
- создаёт fix step;
- передаёт developer agent исходную задачу, ссылку на PR и список замечаний;
- developer agent исправляет PR;
- система запускает повторный review;
- предыдущие итерации остаются в истории run.

Если developer не может исправить замечание без решения человека, run переходит в `blocked`, а Manager Bot задаёт владельцу вопрос с вариантами.

### Поведение при approval

Если reviewer поставил approval:

- система фиксирует reviewer approval;
- пишет краткое summary в Mattermost thread;
- показывает владельцу кнопки "Принять", "Запросить изменения", "Запустить ещё review";
- не делает merge без явного разрешения владельца, если это не настроено отдельно.

## Состояния run

Минимальная state machine:

- `created`;
- `queued`;
- `developer_running`;
- `developer_failed`;
- `pr_opened`;
- `review_running`;
- `changes_requested`;
- `fix_running`;
- `approved_by_reviewer`;
- `waiting_owner`;
- `owner_approved`;
- `owner_rejected`;
- `merged`;
- `blocked`;
- `cancelled`.

## Управление retry

Для review loop нужны настройки:

- минимальное число review: 1;
- максимальное число review: 3;
- что считается request changes;
- кто может остановить цикл;
- какие комментарии передаются developer agent;
- сохранять ли все попытки в Mattermost thread;
- когда эскалировать человеку.

Retry должен быть детерминированным. Система не должна бесконечно запускать агентов. По умолчанию:

- developer запускается один раз;
- reviewer запускается после PR;
- fix loop допускается до трёх раз;
- после трёх request changes run блокируется и ждёт человека.

## Хранение промптов

Для каждого agent profile нужны:

- основной prompt;
- prompt для исправления замечаний;
- prompt для review;
- optional system instructions;
- optional repository-specific overlay.

Промпты должны версионироваться. Изменение prompt должно быть видно в истории run.

Prompt template должен иметь:

- название;
- роль;
- версию;
- текст;
- список переменных;
- автора изменения;
- дату изменения;
- статус: draft/active/archived;
- список flow, где он используется.

Для MVP можно хранить prompt templates в базе данных. Если позже нужна разработка через PR, templates можно вынести в Git-репозиторий и синхронизировать.

## Работа с GitHub

Система должна уметь:

- создавать branch;
- открывать PR;
- читать diff;
- читать review comments;
- оставлять inline comments;
- ставить approval или request changes;
- обновлять PR после исправлений;
- отслеживать merge;
- получать webhooks.

GitHub остаётся источником правды для кода, PR, review и merge.

Важно разделять аккаунты:

- human owner account — принимает финальное решение;
- developer bot account — создаёт ветки и PR;
- reviewer bot account — оставляет review;
- service account — настраивает webhook и читает metadata.

В MVP можно использовать один bot account, но модель должна сразу допускать несколько аккаунтов и разные права.

## Настройка репозитория

При добавлении репозитория система должна:

1. Проверить доступ токена.
2. Проверить default branch.
3. Проверить возможность создать branch.
4. Проверить возможность открыть PR.
5. Проверить возможность оставить comment/review.
6. Настроить или проверить webhook.
7. Сохранить safe metadata.
8. Предложить default flow.

Если чего-то не хватает, Mattermost Bot должен показать понятную ошибку и кнопку "Проверить снова".

## Настройка OpenAI-аккаунтов

Администратор должен иметь возможность добавить несколько OpenAI-аккаунтов:

- основной;
- резервный;
- дешёвый для простых задач;
- мощный для сложных review;
- отдельный аккаунт команды или клиента.

Для каждого agent profile выбирается policy: какой аккаунт использовать, какие модели разрешены, какие лимиты применяются.

## Права доступа

Нужны права:

- кто может добавлять репозитории;
- кто может добавлять токены;
- кто может запускать flow;
- кто может запускать конкретного агента;
- кто может approve owner gate;
- кто может остановить run;
- кто может читать логи;
- кто может редактировать prompt templates.

Для MVP можно начать с ролей `admin`, `maintainer`, `developer`, `viewer`. Но модель должна допускать расширение.

## Безопасность

Обязательные правила:

- секреты не пишутся в Mattermost;
- секреты не попадают в prompt;
- агент получает только нужные credentials;
- все действия пишутся в audit log;
- токены имеют минимальный scope;
- пользовательские команды проверяются по ролям;
- owner approval отделён от agent approval;
- bot approval и human approval различаются.

Отдельно важно:

- не передавать OpenAI raw key агенту как текст;
- не отправлять токены в Mattermost thread;
- не сохранять сырой prompt с секретами;
- маскировать env и command output;
- ограничивать рабочий каталог агента;
- логировать факт использования секрета, но не значение.

## Ошибки и блокеры

Система должна уметь явно показывать:

- agent failed;
- tests failed;
- GitHub token invalid;
- OpenAI quota exceeded;
- repository access denied;
- merge conflict;
- reviewer requested changes;
- human decision required;
- flow limit reached.

Для каждого блокера нужна кнопка следующего действия: retry, сменить token, запросить решение, остановить run, открыть PR, показать summary.

## MVP

Минимальная первая версия:

- подключить Mattermost;
- создать Manager Bot;
- добавить GitHub repository;
- добавить GitHub bot token;
- добавить OpenAI account;
- создать developer profile;
- создать reviewer profile;
- создать один flow `developer-review-loop`;
- запускать run из сообщения Mattermost;
- открывать PR;
- запускать reviewer;
- повторять fix loop до трёх раз;
- показывать статус в Mattermost thread;
- отдавать финальное решение человеку.

MVP должен закрыть один практический путь: пользователь пишет задачу в Mattermost, система доводит её до PR с review-loop и возвращает владельцу понятное решение.

## Критерий успеха MVP

Пользователь в Mattermost пишет задачу в thread, выбирает repository и flow, после чего система сама:

1. запускает developer agent;
2. получает PR;
3. запускает reviewer agent;
4. при request changes запускает исправление;
5. повторяет review;
6. показывает человеку финальное состояние и кнопки решения.

Если это работает стабильно для одного репозитория и одного flow, систему можно развивать дальше.

## Предлагаемый порядок реализации

1. Поднять Mattermost и Manager Bot.
2. Реализовать slash-команды для добавления repository, GitHub token и OpenAI account.
3. Сделать простую базу данных для repository, credentials metadata, agent profiles, flow и run.
4. Реализовать GitHub Adapter для branch/PR/comment/review.
5. Реализовать Agent Runner для запуска одного Codex-agent profile.
6. Реализовать flow `developer-review-loop`.
7. Реализовать Mattermost thread-карточку run с кнопками.
8. Добавить reviewer loop и fix loop до трёх попыток.
9. Добавить owner decision.
10. Добавить audit log и безопасные статусы.
11. Подключить второй agent profile и второй OpenAI account.
12. Провести dogfooding на одном реальном репозитории.

## Открытые проектные решения

Перед началом разработки нужно выбрать:

- запускать агентов локальными процессами, контейнерами или Kubernetes jobs;
- хранить prompt templates в базе или в Git-репозитории;
- делать отдельного Mattermost Plugin или внешний bot-service;
- использовать один GitHub App или набор PAT/bot tokens;
- хранить секреты в Vault, Kubernetes Secrets или другом хранилище;
- показывать агентов как отдельных Mattermost bot users или как одного Manager Bot с подписью роли;
- разрешать ли auto-merge после owner approval;
- какие логи хранить в системе, а какие оставлять только в runner/runtime.

## Главный принцип

Система должна автоматизировать текущую ручную схему, а не ломать её. GitHub остаётся источником правды по коду и PR, Mattermost становится центром управления, агентские профили и flow заменяют ручное копирование промптов, а владелец сохраняет финальный контроль над принятием результата.
