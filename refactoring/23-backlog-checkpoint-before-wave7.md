# Backlog checkpoint между wave 6 и wave 7

## TL;DR
- Checkpoint после wave 6 выполнен как отдельный управленческий результат, а не как "разберёмся потом по ходу".
- Устаревший backlog старых спринтов `S7`, `S9`, `S13`, `S17`, `S18` очищается: часть issues закрывается как устаревшие или заменённые, часть перепривязывается к waves `8`, `11`, `13`, `14`, `15`, `17`.
- Открытых PR на момент checkpoint нет.
- Первая кодовая очередь wave 7 зафиксирована отдельными issues `#599`, `#600`, `#601`, `#602`.

## 1. Цель checkpoint
Этот checkpoint нужен, чтобы:
- не тащить в кодовые волны старые execution anchors, построенные вокруг legacy-модели;
- явно решить судьбу каждого ещё открытого backlog item;
- открыть wave 7 уже с новой очередью задач, а не со смесью старых спринтов и новых доменов.

## 6. Решения по управлению backlog после checkpoint
После этого checkpoint действуют такие правила:
1. Любая новая кодовая задача должна привязываться к одной из волн `7-17`.
2. Старые sprint-bound execution issues не должны использоваться как shortcut для старта кода в обход волн.
3. Если старый issue сохраняется открытым, у него должен быть явно указан новый домен и новая волна.
4. Если старый issue нельзя безболезненно переписать под новую модель, он закрывается и при необходимости заменяется новым issue.

## 7. Следующий шаг после checkpoint

1. считать backlog checkpoint завершённым;
2. использовать `#599`-`#602` как первую кодовую очередь;
3. открыть первую кодовую доменную реализацию — `Wave 7: Access and accounts`.
