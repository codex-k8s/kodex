# Правила ведения документации в программе рефакторинга

## Канонические зоны
### 1. `refactoring/**`
Рабочий source of truth новой версии платформы до тех пор, пока соответствующий домен не будет перенесён в `docs/**`.

### 2. `docs/**`
Боевой каталог канонической документации продукта после завершения и принятия соответствующего refactoring-среза.

### 3. `docs/deprecated/pre-refactor/**`
Архив старых документов, которые были явно вытеснены новой каноникой.

## Правило вытеснения старых документов
- Нельзя держать два активных документа, которые описывают одну и ту же предметную область по-разному.
- После согласования нового домена:
  - новая версия переносится в `docs/**`;
  - старая версия уходит в `docs/deprecated/pre-refactor/**`;
  - индексы и ссылки обновляются в том же PR.

## Языковая политика
- Документация и PR на русском языке.
- Там, где мысль можно выразить по-русски без потери точности, используется русский язык.
- Английский оставляется только для:
  - имён файлов, каталогов, API и протоколов;
  - provider-native сущностей и полей, если важно не потерять точный технический смысл;
  - model ids, env names, code identifiers.
- Смешивание русского текста с лишними англицизмами считается дефектом документа.

## Политика статусов и согласования
- Документ, который публикуется в PR на принятие, должен отражать целевое финальное состояние.
- Для таких документов не используется режим "на согласовании" внутри самого документа.
- Merge PR считается фактом согласования и фиксирует:
  - принятие текста;
  - принятие frontmatter-статуса;
  - принятие связанных решений и ссылок.

## Правило frontmatter для approval PR
- Для документов, которые входят в approval PR, допускается сразу указывать:
  - `status: active`;
  - `approvals.status: approved`.
- Источником факта согласования считается merge этого PR, а не дополнительная ручная правка документа после merge.

## Политика индексов и шумоподавления
- Каждый новый документ сразу добавляется в индекс своей зоны.
- Исторические reference-пакеты должны иметь явную пометку, что они reference-only.
- Документы без роли в текущей канонике не переносятся вперёд механически; они либо архивируются, либо переоформляются в новый домен.

## Минимальный ритуал при завершении доменного среза
1. Утвердить refactoring-версию документации.
2. Перенести новую канонику в `docs/**`.
3. Архивировать старые конфликтующие документы.
4. Обновить индексы.
5. Обновить backlog и карту зависимостей.
