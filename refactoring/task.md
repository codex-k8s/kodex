# Канонический мандат рефакторинга `kodex`

Этот документ — главный источник правды для программы рефакторинга. Он описывает не одну локальную волну, а целевую платформу целиком, включая enterprise-заделы, которые должны быть заложены уже в первой версии и не должны потеряться при дальнейшей декомпозиции.

## 1. Неподвижные рамки

### 1.1. Обратная совместимость не требуется
Платформа нигде ещё не используется, поэтому:
- обратную совместимость сохранять не нужно;
- нерелевантный код и документацию нужно удалять, а не тащить вперёд;
- то, что важно только для истории, переносится в `deprecated` с индексом, а не остаётся активной каноникой.

### 1.2. Рефакторинг идёт сверху вниз
Порядок приоритета такой:
1. бизнес-модель и каноническая документация;
2. сервисные и data-заделы;
3. UX и операторские сценарии;
4. runtime/deploy/bootstrap;
5. волны реализации.

### 1.3. Enterprise target фиксируется сразу
Даже если реализация делается поэтапно, документация должна сразу описывать:
- многоорганизационную модель;
- каталоги пакетов;
- контур серверов и кластеров;
- billing;
- release policy;
- automation rules.

Иначе первые волны реализации зацементируют слишком узкую модель и потом заблокируют расширение.

## 2. Основа, которая остаётся неизменной

### 2.1. Kubernetes-first runtime
Платформа работает в Kubernetes и управляет Kubernetes-first execution model:
- под каждую задачу поднимается slot;
- slot в первой версии — отдельный namespace;
- в `code-only` режиме slot содержит pod агента и минимальный рабочий контур;
- в `full-env` режиме slot поднимает рабочее окружение проекта и pod агента с нужным доступом;
- в enterprise-расширении slot может быть не namespace, а nested-cluster.

Для ускорения работы:
- допустим prewarm slots;
- при reuse slot платформа должна уметь переключать ветку, чистить state, выполнять миграции и загружать фикстуры перед новой сессией.

### 2.2. Provider-first рабочая модель
Платформа не подменяет GitHub/GitLab как систему рабочих артефактов.

Базовые принципы:
- работа строится вокруг `Issue`, `PR/MR`, комментариев, mentions и provider relationships;
- review `PR/MR` остаётся у провайдера;
- platform-owned состояния — это orchestration, runtime, policy, audit, acceptance и projections;
- все provider-операции проектируются через интерфейсы, где GitHub — первая реализация, GitLab — следующая.

### 2.3. Agent-manager как центр управления
У платформы есть быстрые экземпляры `agent-manager`, через которых:
- пользователь общается с системой через фронтенд, текст и голос;
- обрабатываются mentions и запросы из GitHub/GitLab;
- выбираются flow, роли, шаблоны и следующий шаг;
- запускаются role-агенты и обязательные проверки.

Быстрый `agent-manager` по умолчанию использует дешёвую и быструю модель класса `GPT-5.3-Codex-Spark`, но выбор модели должен быть настраиваемым.

### 2.4. Role-driven flow model
В платформе есть:
- flow;
- stage;
- role;
- `stage role binding`;
- acceptance machine.

Эта модель остаётся центральной и в дальнейшем расширяется automation rules, release policy и пакетными контрактами.

Flow, профили ролей и шаблоны промптов являются платформенной конфигурацией, канонически хранятся в БД и редактируются через UI с версиями, аудитом и policy-проверками.

Репозиторий платформы содержит только установочные фикстуры:
- начальные flow;
- встроенные профили ролей;
- встроенные `work`- и `revise`-шаблоны промптов;
- seed-метаданные для первичной загрузки БД при установке или обновлении платформы.

Агенты в первой версии не редактируют flow, профили ролей и шаблоны промптов. Будущий self-improve для этих объектов должен идти через отдельный контур согласования и публикации новой версии в БД, а не через чтение конфигурации из репозитория при каждом запуске.

### 2.5. Frontend и operator console
Фронтенд полностью переосмысляется:
- стек остаётся;
- UI-библиотека — `PrimeVue`;
- главный рабочий режим — командный центр с чатом и голосом;
- отдельные рабочие пространства нужны для `Issue`, `PR/MR`, `run`, `job` и slot;
- отдельный настроечный контур управляет flow, ролями, каталогами пакетов, доступами, внешними аккаунтами, automation rules и другими policy-объектами.

## 3. Обязательные новые направления wave 5.1

### 3.1. Система плагинов и пакетная платформа
Платформа должна иметь полноценную пакетную модель.

Базовая идея:
- есть встроенные пакеты;
- есть внешние пакеты из каталога;
- есть пользовательские пакеты из произвольных источников;
- пакет может быть бесплатным или платным;
- пакет и конкретная версия могут быть верифицированы или не верифицированы;
- источником правды для пакета является репозиторий-источник: встроенный, внешний каталог или пользовательский источник.

Виды пакетов:
- `plugin package` — исполняемое расширение платформы;
- `guidance package` — пакет руководящей документации, шаблонов и правил для агентов.

`Plugin package` должен описываться через manifest в репозитории и включать:
- идентичность, описание, локализованные названия и ссылки на изображения;
- вид пакета;
- версию;
- verification и коммерческий статус;
- требуемые platform APIs;
- обязательные и желательные permissions;
- schema для secret inputs с локализованными названиями, описаниями и типами значений;
- Dockerfile;
- Kubernetes manifests;
- другие runtime-зависимости, необходимые для запуска.

Плагин должен запускаться в Kubernetes-контуре платформы того, кто его установил, и может быть написан на любом языке и стеке.

Примеры, которые сразу должны идти через пакетную платформу:
- Telegram-плагин для согласований;
- Telegram-плагин для обратной связи;
- внешние интеграции для уведомлений, согласований и обратной связи.

### 3.2. Система руководящей документации и каталог пакетов руководящей документации
Руководящая документация должна перестать быть неразделимой частью одного монорепозитория.

Нужно сразу заложить:
- отдельные репозитории руководящей документации;
- каталог пакетов руководящей документации;
- модель импорта в проект, репозиторий, сервис или саму платформу;
- работу агента только с локально доступной документацией без векторного хранилища в первой версии.

В частности, должны быть возможны отдельные репозитории для:
- общих руководящих правил;
- правил разработки по направлениям/стекам;
- правил проектирования;
- правил ведения продуктовой документации;
- других руководящих пакетов.

Ключевой принцип:
- пакет руководящей документации живёт в репозитории-источнике;
- в рабочий контур он попадает как локальная копия или импорт;
- self-improve задача может запускаться против самого репозитория руководящей документации, а не только против проектного монорепозитория.

Отдельно от руководящей документации существует проектная документация: продуктовые документы, описания сервисов, архитектурные решения, runbook и договорённости конкретного проекта. Она может жить в одном или нескольких репозиториях проекта и не должна превращаться в пакет руководящей документации только потому, что агенту нужно её читать.

Для проекта с несколькими репозиториями `services.yaml` должен описывать:
- какие репозитории входят в проект;
- где лежит общая продуктовая документация;
- где лежит документация конкретного сервиса;
- какие сервисы зависят от документации других сервисов;
- какие источники документации доступны агенту только для чтения, а какие можно менять в рамках задачи;
- в какие локальные пути рабочего контура эти источники загружаются.

Рабочий контур агента в slot должен получать не только кодовой репозиторий, но и полный разрешённый набор локальных источников: код, проектную документацию, документацию зависимых сервисов и пакеты руководящей документации. Flow, роли и шаблоны промптов передаются агенту как отрендеренный платформой контекст из БД, а не как отдельный набор файлов в workspace.

### 3.3. Организации, группы пользователей и гибкая модель доступа
Платформа должна поддерживать:
- owner-организацию;
- клиентские организации;
- организации внешних исполнителей;
- будущие SaaS-организации.

Пользователь может:
- состоять в нескольких организациях;
- состоять в нескольких группах;
- получать права напрямую и через membership в group или organization;
- иметь разные роли и права в разных контурах.

Группы бывают:
- глобальные;
- организационные;
- в будущем допускаются проектные и специальные группы, если они не ломают общую модель.

Access model должна поддерживать:
- inheritance;
- явные overrides;
- приоритеты правил;
- разграничение по пользователям, группам, организациям, ролям агентов и другим policy-субъектам.

### 3.4. Серверы, кластеры и контур серверов и кластеров
Мы стартуем с одного сервера и одного Kubernetes-кластера, но документация и код больше не должны считать эту топологию единственно возможной.

Нужно сразу заложить:
- реестр серверов;
- реестр Kubernetes-кластеров;
- связи между организациями, проектами, репозиториями и допустимыми серверными и кластерными контурами;
- placement policy;
- подготовку к multi-zone и delegated clusters;
- возможность закреплять проект, репозиторий или отдельный тяжёлый сервис за конкретным кластерным контуром.

### 3.5. Биллинг и cost accounting
Платформа должна сразу проектироваться так, чтобы:
- учитывать затраты на compute, storage и runtime;
- учитывать внешние провайдеры и их стоимость;
- учитывать использование пакетов и в будущем комиссии каталога;
- раскладывать расходы по организациям, проектам и при необходимости по другим контурам;
- формировать основу для invoice и коммерческого учёта.

В enterprise target нужно предусмотреть:
- рублёвый платёжный контур через российские эквайринги;
- валютный контур через `Stripe`;
- выставление счетов организациям, в том числе с разбиением по проектам.

### 3.6. Branch rules, release policy и связь с flow
Платформа должна различать:
- provider-native branches и tags;
- release branches;
- release line или release train;
- release policy;
- политика выкладки;
- release task и release flow.

Нужно заложить модель, где:
- задачи и `PR/MR` попадают в release line;
- релизная ветка и связанные проверки живут у провайдера;
- платформа ведёт policy, gates, стратегию выкладки и операторскую логику;
- можно делать прямой rollout, staged rollout, canary и rollback.

### 3.7. Автозапуск flow по cron и автоматизация по триггерам
Платформа должна уметь:
- запускать flow по расписанию;
- создавать задачи по расписанию;
- реагировать на внешние триггеры;
- запускать специальные agent roles по alerts и operational signals.

Примеры:
- ночной flow для анализа production logs и метрик;
- flow по поиску slow queries;
- триггер на входящий alert от `Prometheus`;
- запуск расследования по превышению порога `500`-ошибок.

Это должно проектироваться как обязательная способность платформы, а не как ad-hoc cron script рядом с платформой.

## 4. Минимальный объём первой версии и enterprise target

| Направление | Минимальный объём первой версии | Enterprise target |
|---|---|---|
| Организации и группы | owner-организация, клиентские организации, участие пользователей и групп, scoped permissions | SaaS-модель с изоляцией по организациям, delegated admins, сложные inheritance/override policies |
| Пакетная платформа | встроенные пакеты, пользовательские источники, локальная установка плагинов и пакетов руководящей документации, признак верификации | marketplace, verified package versions, publisher profiles, paid/free packages, revenue share |
| Контур серверов и кластеров | один production-кластер с явным слоем реестра и политикой размещения | несколько кластеров, несколько серверов, привязка кластера к организации или проекту, мультизональность |
| Учёт затрат | записи затрат, распределение использования, сводки для биллинга | invoices, payment providers, биллинг каталога пакетов, automated charging |
| Релизная политика | релизная политика, правила релизных веток, релизная линия, задел для canary и staged rollout | полные программы релизов, политики выкладки с учётом организаций, сложные релизные линии |
| Автоматизация | cron rules и минимальные event triggers | шины событий, переиспользуемые шаблоны автоматизации, сложные графы триггеров |
| Пакеты руководящей документации | импорт из отдельных репозиториев в локальный рабочий контур | marketplace guidance packages, платные и бесплатные bundles, managed updates |
| Flow, роли и шаблоны промптов | каноническое хранение в БД, UI-редактирование, версии, аудит, установочные фикстуры в репозитории платформы | self-improve через согласуемые изменения версий, расширенные политики публикации и делегированное администрирование |
| Проектная документация | ссылки на источники в `services.yaml`, загрузка в workspace и возможность связанного набора `PR/MR` | расширенные политики доступа, проверки доступности и согласованные изменения в нескольких репозиториях |

## 5. Архитектурные следствия

### 5.1. Новые или расширяемые owner-domains
Wave 5.1 означает, что в целевой архитектуре должны быть зафиксированы:
- tenancy-aware `access-manager`;
- расширенный `project-catalog`;
- `package-hub`;
- `fleet-manager`;
- `billing-hub`;
- расширенный `agent-manager` с automation rules;
- расширенный governance-контур для release policy и unattended automation.

### 5.2. Что остаётся во владении провайдера
Даже после этих расширений провайдер остаётся владельцем:
- branches и tags;
- `Issue`, `PR/MR`, comments и review state;
- repository storage.

Платформа не должна плодить внутренние дубликаты там, где достаточно policy, mirror и orchestration.

## 6. Что нужно сделать прямо сейчас в документации
Сейчас, до старта wave 6, нужно:
1. актуализировать `refactoring/task.md` и индекс программы;
2. добавить канонику wave 5.1;
3. обновить wave 0-5 так, чтобы пакетная платформа, tenancy, контур серверов и кластеров, billing, release policy и automation rules уже были видны в продуктовой, архитектурной, data, governance и UX-модели;
4. не пытаться детально расписать runtime/deploy ещё раз здесь — это задача wave 6;
5. не тащить это в код до завершения docs-only синхронизации.

## 7. Что нельзя потерять после сжатия контекста
Если контекст сессии будет сокращён, обязательными для восстановления картины остаются следующие идеи:
1. пакетная платформа = общий слой для плагинов и пакетов руководящей документации;
2. организации и группы закладываются сразу, даже если MVP прост;
3. один сервер и один cluster — это только стартовая операционная конфигурация;
4. billing и cost accounting нужно проектировать сейчас, а не после MVP;
5. release branches, release policy, release trains и стратегия выкладки — часть канонической модели;
6. automation rules по cron и trigger-driven events — часть flow model;
7. пакеты руководящей документации должны жить в репозиториях-источниках и попадать в рабочий контур локально;
8. flow, роли и шаблоны промптов канонически хранятся в БД, редактируются через UI и загружаются из репозиторных фикстур только при установке или seed-обновлении;
9. проектная документация описывается источниками в `services.yaml` и загружается агенту локально вместе с кодом и руководящими пакетами;
10. всё это должно быть отражено в волнах 0-5 до старта wave 6 и волн реализации.
