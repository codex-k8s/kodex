## Готовность к началу

- Доменный пакет документации принят.
- Целевой каталог нового сервиса не пересекается со старым `deprecated/**`.
- Известны будущие пути proto/OpenAPI/миграций или создана задача на их создание.
- Все команды изменения проектируются с идемпотентным ключом и ожидаемой версией агрегата.
- Секреты проектируются через ссылку на секрет, а не через хранение значения в БД.

## Критерии завершения

- В новом коде есть `access-manager` как сервис-владелец домена.
- Есть собственный контур БД, миграций и typed domain models.
- Организации, пользователи, группы, членство, allowlist, поставщики внешних аккаунтов, внешние аккаунты, действия доступа и правила доступа реализованы без произвольного JSON как основной модели.
- Есть outbox для доменных событий `access.*`.
- Есть API для `CheckAccess`, `ExplainAccess`, `BootstrapUserFromIdentity` и управления внешними аккаунтами.
- Есть след аудита для административных изменений и решений доступа.
- Документация и карты Issue синхронизированы с фактической реализацией.

## Зависимости

- Базовая инфраструктура разработки новых сервисов должна создаваться с нуля, без переноса старого `control-plane`.
- Операции провайдера в следующих волнах должны использовать ссылки на внешние аккаунты из `access-manager`.
- Пользовательский интерфейс может стартовать с минимального пути чтения, но не должен читать БД напрямую.

## Риски и буферы

| Риск | Митигирование |
|---|---|
| Одноорганизационная модель зацементирована в коде | Все сущности сразу имеют модель организации и области применения. |
| Секреты попали в БД или аудит | Ссылка на секрет, маскированная диагностика и проверки безопасности. |
| `provider-hub` начал владеть политикой аккаунта | Контракт: `provider-hub` получает разрешённое использование аккаунта у `access-manager`. |
| Проверки доступа размазаны по сервисам | Все сервисы используют `CheckAccess` и не создают локальные правила. |
