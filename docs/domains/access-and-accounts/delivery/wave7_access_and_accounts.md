## Инфраструктурный бэклог перед следующим сервисом

До создания следующего доменного сервиса общий gRPC-контур вынесен из `access-manager` в `libs/go/grpcserver`.

В общий пакет должны перейти только инфраструктурные части без доменных зависимостей:
- сборка `grpc.Server` из типизированной конфигурации;
- восстановление после panic;
- лимит активных RPC;
- таймаут выполнения RPC;
- keepalive-политика, `MaxConcurrentStreams`, ограничение размера сообщений;
- базовые Prometheus-метрики unary RPC;
- расширяемая проверка вызывающей стороны без привязки к домену `access-manager`.

В общий пакет не переносить преобразования `proto <-> domain`, маппинг доменных ошибок, конкретные доменные типы и обработчики `AccessManagerService`. `access-manager` становится первым потребителем общей библиотеки, а следующий сервис не должен копировать локальные gRPC-помощники.

## Готовность к началу

- Доменный пакет документации принят.
- Целевой каталог нового сервиса не пересекается со старым `deprecated/**`.
- Машинно-проверяемые контракты `proto` и AsyncAPI созданы как стабильный `v1` на весь согласованный объём внутреннего доменного API.
- Все команды изменения проектируются с идемпотентным ключом и ожидаемой версией агрегата.
- Секреты проектируются через ссылку на секрет, а не через хранение значения в БД.

## Контрактная готовность и бэклог реализации

Стабильный контракт `v1` фиксирует целевой внешний и межсервисный API домена. Это не означает, что весь контракт уже реализован в коде текущего среза. Реализация закрывается по запланированным Issue домена до того, как соответствующие операции станут обязательной зависимостью других сервисов или пользовательского интерфейса.

## Критерии завершения

- В новом коде есть `access-manager` как сервис-владелец домена.
- Есть собственный контур БД, миграций и typed domain models.
- Есть промышленные манифесты, отдельное задание миграций, секреты времени выполнения, health/readyz/metrics и проверочный путь выкладки сервиса.
- Организации, пользователи, группы, членство, allowlist, поставщики внешних аккаунтов, внешние аккаунты, действия доступа и правила доступа реализованы без произвольного JSON как основной модели.
- Есть outbox для доменных событий `access.*`.
- Есть API для `CheckAccess`, `ExplainAccess`, `BootstrapUserFromIdentity` и управления внешними аккаунтами.
- Есть след аудита для административных изменений и решений доступа.
- Документация и карты Issue синхронизированы с фактической реализацией.

## Зависимости

- Базовая инфраструктура разработки новых сервисов должна создаваться с нуля, без переноса старого `control-plane`.
- Операции провайдера в следующих волнах должны использовать ссылки на внешние аккаунты из `access-manager`.
- Пользовательский интерфейс может стартовать с минимального пути чтения, но не должен читать БД напрямую.

## Риски и буферы

| Риск | Митигирование |
