# Пакетная платформа

## Назначение

Домен описывает плагины, пакеты руководящей документации, клиентские подключения к магазинам пакетов, каталоги, версии, установку, верификацию, права доступа и коммерческий статус пакетов.

## Связанные каталоги

- `docs/catalogs/package-store/`
- `docs/catalogs/plugins/`
- `docs/catalogs/guidance-packages/`

## Ключевые решения

- Каталог установленных пакетов хранится локально в платформе.
- Каталог доступных пакетов получает данные через клиентские подключения к магазинам пакетов.
- Магазин пакетов является устанавливаемым пакетом из репозитория-источника.
- Пакетная платформа не владеет магазином как бизнес-системой: она выступает клиентом магазина, хранит подключение, синхронизирует доступные пакеты и управляет локальными установками.
- Авторский магазин `kodex` живёт в приватном репозитории `github.com/codex-k8s/kodex-package-store` и не вшивается в ядро платформы.
- Сайт и пользовательская документация платформы поставляются отдельным пакетом из приватного репозитория `github.com/codex-k8s/kodex-package-platform-site`. В этот пакет входят материалы для пользователей платформы: API, разработка пакетов, добавление пакетов в магазин и каталог.
- В основном репозитории оба приватных источника подключены как необязательные submodule внутри `packages/**`, чтобы обычный clone не требовал доступа к ним.
- Submodule в рабочем репозитории используются только для platform-owned source repos и публичных пакетов правил/шаблонов.
- Установленные пакеты, клиентские пакеты и платные пакеты не должны зависеть от submodule рабочего репозитория: их поставка идёт через пакетную платформу по зафиксированному `tag` или `commit`, а платные пакеты — через прокси платформы.
