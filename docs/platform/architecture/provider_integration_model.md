# Модель интеграции с провайдерами

## Кратко

GitHub является первой реализацией контура интеграции с провайдерами, GitLab должен лечь в ту же архитектурную границу. Агенты в слотах могут работать с нативными артефактами провайдера через `gh` или нативный API, а быстрый agent-manager и платформенные операции идут через `platform-mcp-server` и `provider-hub`. Пользовательский интерфейс и приёмка работают по зеркальным проекциям и не читают провайдера на каждый экран.

## Контур интеграции с провайдерами

Контур интеграции с провайдерами покрывает:
- репозитории кода и рабочих артефактов;
- репозитории проектной документации;
- репозитории пакетов и руководящей документации;
- релизные ветки и теги;
- операционное состояние авторизации, лимитов и блокировок по внешним аккаунтам, зарегистрированным в `access-manager`;
- будущие внешние системы, где платформа не хранит истину сама.

`provider-hub` является сервисом-владельцем интеграции с провайдерами. Он изолирует особенности GitHub/GitLab и отдаёт остальным сервисам нормализованные контракты. Канонический реестр внешних аккаунтов, область их применения и политика использования находятся в `access-manager`.

## Категории операций

| Категория | Кто вызывает | Как проходит |
|---|---|---|
| Событие webhook | Провайдер | `integration-gateway` проверяет запрос на границе, `provider-hub` сохраняет inbox и нормализует событие. |
| Каноническое чтение | Сервис-владелец или приёмка | `provider-hub` читает провайдера с учётом лимитов и разрешения на использование внешнего аккаунта. |
| Операция провайдера | Agent-manager или платформенный сервис | Через `platform-mcp-server` и `provider-hub`, если операция платформенная или требует политики и аудита. |
| Работа агента в слоте | Ролевой агент | Через `gh`, `glab` или нативный API в слоте по профилю роли. |
| Сверка | `worker` по поручению `provider-hub` | Инкрементальный курсор, окно перекрытия, приоритеты горячих, тёплых и холодных сущностей. |

## Нормализованные объекты

| Объект | Минимальный смысл |
|---|---|
| `ProviderAccountRuntimeState` | Операционное состояние внешнего аккаунта у провайдера: статус авторизации, лимиты, доступность операций и последние ошибки. |
| `ProviderRepository` | Внешний репозиторий, владелец, видимость, идентификатор провайдера, URL и привязка к проекту. |
| `ProviderWorkItem` | Нормализованный `Issue` или `PR/MR` с идентификатором провайдера, номером, состоянием, заголовком, метками, назначенными пользователями, вехой и проектными полями. |
| `ProviderComment` | Комментарий, упоминание или сигнал review, связанный с рабочим артефактом. |
| `ProviderRelationship` | Связь между `Issue`, `PR/MR`, последующей задачей, блокировкой, релизом или артефактом. |
| `ProviderLimit` | Текущее известное состояние лимитов, время сброса, класс ограничения и затронутый аккаунт. |
| `ProviderOperation` | Запись внешней операции, инициатор, аккаунт, цель, результат и ошибка. |

## Синхронизация проекций

### Входящий журнал webhook

Webhook является быстрым входом, но не единственной гарантией актуальности.

Обязательные правила:
- входящее событие сначала сохраняется во входящий журнал;
- дедупликация выполняется по delivery id или аналогу провайдера;
- нормализация и пересчёт проекции идут асинхронно;
- ошибка обработки не удаляет исходные данные до истечения срока хранения;
- каждый пересчёт публикует доменное событие для потребителей.

### Инкрементальная сверка

Инкрементальная сверка обязательна, потому что webhook может потеряться или прийти с задержкой.

Правила:
- хранить `sync_cursor` по области синхронизации: репозиторий, тип артефакта или другая область провайдера;
- использовать окно перекрытия для пограничных изменений;
- различать горячие, тёплые и холодные сущности;
- при дефиците лимитов сохранять бюджет на горячие сущности;
- повышать `drift_status`, если проекция могла устареть.

### Сигнал от агента в слоте

Если агент в слоте создал или изменил артефакт через `gh` или API, он передаёт платформе идентификатор провайдера, URL или `owner/repo/number` как ускоряющий сигнал. Этот сигнал не заменяет webhook и сверку, а только помогает быстрее обновить проекцию.

## Лимиты и учёт `gh`

Операции через `provider-hub` учитываются детерминированно: платформа знает аккаунт, операцию, цель, заголовки ответа и итог.

Для прямой работы агента в слоте через `gh` действует приближённый учёт:
- до и после агентного запуска снимается `gh api rate_limit`;
- для высокообъёмных действий агент отправляет промежуточные снимки через MCP;
- если используется `gh api`, агент может передать заголовки ответа, полученные через `gh api --include`;
- высокоуровневые команды `gh issue` и `gh pr` не считаются полным источником телеметрии.

Целевое усиление: операции, где нужен строгий лимитный бюджет, должны идти через прокси провайдера или MCP-инструменты платформы.

## Нативные поля провайдера

Платформа использует нативные поля провайдера там, где они полезны человеку и не требуют постоянного переписывания быстро меняющегося состояния среды исполнения.

Допустимые поля:
- type;
- метки;
- связи;
- комментарии и упоминания;
- состояние;
- состояние review;
- назначенные пользователи;
- milestone;
- проектные поля;
- ветки и теги для релизной политики.

Не размещать в нативных полях провайдера:
- текущее состояние слота;
- внутренние retry-счётчики;
- полный лог среды исполнения;
- технические детали задания;
- быстро меняющиеся диагностические данные.

## Сначала GitHub, затем GitLab

Первая реализация:
- GitHub webhook;
- `Issue`, `PR`, комментарии, упоминания и review;
- использование GitHub bot/user аккаунтов, зарегистрированных в `access-manager`;
- `gh` в слоте;
- обработка лимитов;
- зеркало провайдера и сверка.

Архитектурные требования для следующего провайдера:
- в доменах нет прямых GitHub REST/GraphQL путей;
- нормализованная модель не содержит обязательных полей, доступных только в GitHub, без резервного варианта;
- `provider-hub` скрывает различия API;
- пользовательский интерфейс использует нормализованные проекции.
