# Инструкции для агентов в `docs/platform/**`

## Назначение

`docs/platform/**` хранит сквозную канонику платформы: продуктовую модель, целевую архитектуру, общие правила поставки и эксплуатации.

## Перед изменениями

- Читать `docs/AGENTS.md`.
- Читать `docs/platform/README.md`.
- Читать релевантные документы `refactoring/*.md`, если изменение связано с программой рефакторинга.
- Читать `docs/templates/index.md` и выбирать шаблон под тип создаваемого документа.

## Куда класть документы

- `docs/platform/product/**` — сквозные продуктовые требования, ограничения, роли пользователей и сценарии платформы.
- `docs/platform/architecture/**` — общая архитектура, C4, общая модель данных, межсервисные контракты и системные решения.
- `docs/platform/delivery/**` — общие правила поставки, приёмки, релизной политики и управления изменениями.
- `docs/platform/ops/**` — общие правила эксплуатации, мониторинга, резервного восстановления и действий при инцидентах.

## Что не класть в `docs/platform/**`

- Документы конкретного домена. Их место — `docs/domains/<domain>/**`.
- Карты `Issue/PR ↔ документы`. Их место — `docs/delivery/issue-map/**`.
- Каталоги плагинов, пакетов руководящей документации, ролей и шаблонов. Их место — `docs/catalogs/**`.
- Исходные исследования без принятого решения. Их место — `docs/research/**`.
- Старые материалы из `deprecated/**` без переработки по актуальной канонике.

## Правила содержания

- Сквозной документ должен задавать рамку для нескольких доменов или всей платформы.
- Если текст начинает описывать детали одного домена, нужно вынести их в доменный документ и оставить ссылку.
- Не фиксировать временные решения как платформенную канонику, если они нужны только для текущего рабочего среза.
- В теле документов `docs/platform/product/**` не указывать номера волн, Issue, PR и другие динамические ссылки. Frontmatter-поля связей остаются обязательным местом для таких ссылок, а в основном тексте продуктовые документы должны ссылаться на устойчивые возможности, домены и контуры.
