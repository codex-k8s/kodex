# Ролевая модель координации

## Назначение

Этот документ задаёт драфт активной модели работы в новом окружении, где задачи выполняются не постоянной сеткой из шести разработчиков и шести ревьюверов, а ролями. Роль описывает ответственность в процессе поставки; конкретный исполнитель в приложении может меняться от задачи к задаче.

Правила ниже не являются финальной конституцией процесса. Новые агенты должны уточнять их по мере обкатки приложения и фиксировать изменения в PR.

## Текущая точка перехода

- Фаза параллельной разработки шестью доменными агентами завершила базовые срезы перезапуска MVP: ручной bootstrap/adoption, контур provider-native, обычный `AgentRun`, контекст интерфейса владельца и оператора, готовность ручной выкладки и базовый контур governance.
- Ручная MVP-выкладка свежего `main` `9cca0f4c` выполнена по задаче `#1111`: backend-кольца, `staff`, `web` и `web-public` применены штатным путём, эксплуатационная приёмка прошла, публичный UI закрыт OAuth, неподписанный GitHub webhook отклоняется без OAuth redirect.
- Самовыкладка не является критическим путём MVP. Она остаётся отдельной возможностью и не должна блокировать ручную выкладку, проверку платформы на самой себе и обычный пользовательский сценарий.
- Следующий фокус — сквозная проверка обычного MVP-сценария: проект и репозиторий, проверенный `services.yaml`, запуск `AgentRun`, активность, приёмка, артефакт провайдера, решение владельца, governance-сводка и отображение в `web-console`.

## Как выполнялась работа до перехода

Предыдущая рабочая схема была временной и оптимизированной под быстрый прогресс в текущем репозитории:

- manager вёл общий статус, подтягивал `main`, анализировал слитые PR, создавал Issues и выдавал короткие промпты исполнителям и ревьюверам;
- шесть доменных исполнителей работали в отдельных клонах репозитория и закрывали параллельные срезы по крупным доменам;
- у каждого доменного исполнителя был отдельный ревьювер, который проверял PR по задаче, документации, гайдам и live-safe правилам;
- SRE-задачи фактически выполнялись отдельным рабочим потоком вокруг bootstrap, Kubernetes, Kaniko, registry, Secrets, ingress, сертификатов и эксплуатационной приёмки;
- вся существенная приёмка закреплялась в GitHub Issues, PR, issue-map и delivery-документах, а не только в чате.

Эта схема остаётся историческим описанием выполненной фазы. В новом приложении не нужно воспроизводить постоянные пары “шесть разработчиков и шесть ревьюверов”; вместо этого задачи назначаются ролям.

## Активные роли

| Роль | Ответственность |
|---|---|
| Manager | Ведёт общий план, создаёт GitHub Issues, раскладывает зависимости, решает порядок запуска ролей, следит за тем, чтобы работа двигала MVP-сценарий, а не набор несвязанных задач. |
| Architect | Проверяет границы доменов, контракты, владение данными, события, миграции, совместимость обновления live-контура и соответствие `docs/platform/**` и `docs/domains/**`. |
| Developer | Делает кодовый срез по Issue: сервисы, контракты, миграции, тесты, генерация, манифесты и связанные технические документы. |
| Docs | Делает документационный срез по Issue: продуктовые документы, архитектурные описания, runbook, карты связей, инструкции и синхронизацию документации с кодом. |
| Reviewer | Проводит проверку PR: сверяет задачу, код, документы, тесты, миграции, безопасность данных и live-safe правила. Замечания оставляет в PR; если блокеров нет, ставит approve. |
| QA | Проверяет приёмку: Go-проверки, интеграционные runner-ы, UI-сценарии, fixture-наборы, регрессии и соответствие критериям готовности задачи. |
| SRE | Отвечает за путь выкладки, Kubernetes, registry/Kaniko, Secrets, готовность, задания, сертификаты, публичный ingress, эксплуатационную приёмку, runbook и безопасную диагностику live-блокеров. |

## Рабочий цикл

1. Manager формулирует GitHub Issue с целью, границами, артефактами, проверками и критериями готовности.
2. Architect участвует до реализации, если задача меняет доменные границы, контракты, миграции, события, путь выкладки или live-safe порядок.
3. Developer или Docs работает в отдельной ветке, обновляет код или документацию, запускает применимые проверки и открывает PR.
4. Reviewer проверяет PR по задаче, гайдам и архитектуре. Если замечаний нет, ставит approve; если есть блокеры, оформляет конкретные замечания.
5. QA запускает приёмку, когда задача требует runtime, UI, интеграционных или регрессионных проверок.
6. SRE выполняет rollout и live-проверки только по принятому пути выкладки. Ручные `kubectl patch`, ручной SQL, изменение live-секретов вне инструментов выкладки и обход миграций запрещены без отдельного решения владельца.
7. Manager обновляет статус и решает следующий срез после merge и проверки.

## Правила для нового окружения

- Новые задачи не назначать как `агент #1` ... `агент #6`, если пользователь явно не просит совместимость со старой раскладкой.
- Старые номера агентов можно упоминать только как исторический источник выполненных срезов или при разборе старого PR.
- В описаниях задач использовать роли: `manager`, `architect`, `developer`, `docs`, `reviewer`, `qa`, `sre`.
- Вся существенная информация для передачи между окружениями должна попадать в Issue, PR, `docs/delivery/**`, runbook или доменную документацию. Не оставлять критичный статус только в чате.
- Ручная выкладка остаётся основным MVP-путём. Проверки самовыкладки допустимы только как отдельный явный режим и не блокируют обычный MVP-сценарий.
