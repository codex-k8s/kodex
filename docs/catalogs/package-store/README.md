# Магазин пакетов

## Назначение

Раздел описывает будущий магазин пакетов как устанавливаемый пакет, а не как жёстко встроенную часть ядра платформы.

У каждой установки платформы должен быть локальный каталог установленных пакетов и локальный каталог доступных пакетов. Каталог доступных пакетов получает данные через клиентские подключения к одному или нескольким магазинам пакетов.

## Каноническое решение

- Авторский магазин пакетов `kodex` поставляется из приватного репозитория `github.com/codex-k8s/kodex-package-store` как пакет.
- Мы устанавливаем этот пакет в собственную платформу и через него публикуем доступные пакеты.
- Другая установка платформы может использовать наш магазин, свой магазин или только пользовательские источники.
- Отдельный пакет из приватного репозитория `github.com/codex-k8s/kodex-package-platform-site` поставляет сайт платформы и пользовательскую документацию платформы: API, разработка пакетов, добавление пакетов в магазин и каталог.
- В `kodex` оба репозитория объявлены в `.gitmodules` и checkout-ятся внутри `packages/**` после проверки доступа.

## Политика submodule и установки пакетов

- Source submodule пакетов объявляются в `.gitmodules` и размещаются в `packages/**`.
- Пакетная платформа хранит метаданные установленных пакетов в БД.
- Начальные метаданные пакета можно предзаполнить из JSON-манифеста внутри репозитория пакета.
- Пакетная платформа должна получать пакет по зафиксированному `tag` или `commit`.
- Для платных пакетов поставка идёт через прокси платформы, а checkout разрешается только после проверки прав.
- Для downstream-репозиториев отдельный workflow удаляет магазин, сайт и необязательные пакеты, которые не должны попадать в конкретную установку.

## Именование platform-owned пакетов

- Platform-owned package repos именуются по схеме `kodex-package-<name>`.
- Имя должно описывать устанавливаемый пакет, а не проект-потребитель.
- Для сайта и пользовательской документации используется slug `kodex-package-platform-site`.

## Что не делать

- Не вшивать единственный магазин пакетов в ядро платформы.
- Не смешивать публичный сайт платформы с руководящими пакетами для агентов.
- Не считать каталог доступных пакетов источником состояния установки: состояние установки хранится локально в платформе.
